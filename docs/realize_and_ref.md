# The `realize`/`ref` Protocol

This document explains the lazy proxy realization system that is at the core of
Khepri's backend abstraction. Every shape, material, layer, and family flows
through this system to become a backend-specific object.

---

## 1. The Lazy Proxy Pattern

Shapes, materials, layers, and families in Khepri are **proxies** — lightweight
Julia structs that defer backend realization. When a user writes `sphere(xyz(0,
0, 0), 5)`, Khepri does not immediately create a sphere in any backend. Instead
it creates a `Sphere` proxy struct and, depending on the current transaction,
either realizes it immediately or defers it for later.

The lifecycle of a proxy is:

1. **Construction** — The user calls a constructor (e.g., `sphere(...)`)
2. **`after_init`** — Post-construction hook, calls `maybe_realize(s)`
3. **`maybe_realize`** — Dispatches based on current transaction type:
   - `AutoCommitTransaction`: calls `force_realize(b, s)` immediately
   - `ManualCommitTransaction`: pushes `s` into `t.proxies` for later
4. **`realize(b, s)`** — Calls the backend's `b_<name>(b, ...)` operation
5. **`ensure_ref`** — Wraps the raw return value into a `GenericRef{K,T}`
6. **Storage** — The wrapped reference is stored in `refs_storage(b, s)`

Subsequent calls to `ref(b, s)` return the stored reference without
re-realizing.

### `after_init` variants

There are three `after_init` methods:

- **`Proxy`**: calls `maybe_realize(s)`, returns `s`
- **`Shape`**: calls `maybe_realize(s)`, `maybe_collect(s)`, `maybe_trace(s)`,
  returns `s`
- **`UniqueProxy`**: searches existing refs to avoid duplicates — if a proxy
  with the same spec already exists, returns that one instead of creating a new
  realization

## 2. The Reference Type Hierarchy

References are parameterized by backend key `K` and raw reference type `T`:

```
GenericRef{K,T}           (abstract)
├── NativeRef{K,T}        single raw backend reference
├── NativeRefs{K,T}       multiple raw backend references (Vector{T})
└── LocalRef{K,T}         local value that doesn't match T
```

**Defined in**: `Types.jl:153-170`

### `NativeRef{K,T}`

The common case. Wraps a single value of type `T` returned by a `b_*`
operation. For example, `NativeRef{ACADKey, Int64}(42)` wraps an AutoCAD object
handle.

### `NativeRefs{K,T}`

Used when a single shape requires multiple backend objects. For example, a
cylinder might be realized as a tube plus two caps, each returning a separate
reference. The `ensure_ref` function automatically wraps a `Vector{T}` into
`NativeRefs` when the vector has more than one element.

### `LocalRef{K,T}`

A fallback for when `ensure_ref` receives a value that is neither `T` nor
`Vector{T}`. This is rare and typically indicates a non-standard return from a
`b_*` operation.

## 3. Core Functions

All defined in `Shapes.jl` unless noted otherwise.

### `realize(b, s)` — Dispatch realization to the backend

```julia
realize(b::Backend, s::Proxy) = error("Undefined realization for $(typeof(s)) in $(b)")
```

The default throws an error. Concrete methods are generated by `@defproxy` and
`@defshape` (see Section 5). Each generated method calls the appropriate
`b_<name>(b, ...)` function.

### `ref(b, s)` — Get-or-realize

```julia
ref(b::Backend, s::Proxy) =
  let refs = refs_storage(b, s)
    haskey(refs, s) ?
      refs[s] :
      refs[s] = ensure_ref(b, realize(b, s))
  end
```

Checks `refs_storage` for a cached reference. On cache miss, calls `realize`,
wraps the result with `ensure_ref`, stores it, and returns it.

### `ref!(b, s, r)` — Store a reference

```julia
ref!(b::Backend, s::Proxy, r) =
  let refs = refs_storage(b, s)
    refs[s] = ensure_ref(b, r)
  end
```

Wraps `r` via `ensure_ref` and stores it. Used to manually associate a proxy
with a backend reference (e.g., when importing existing objects from a backend).

### `force_realize(b, s)` — Always realize and store

```julia
force_realize(b::Backend, s::Proxy) =
  ref!(b, s, realize(b, s))
```

Unconditionally calls `realize` and stores the result. Unlike `ref`, this
ignores any cached reference.

There is also a multi-backend variant:

```julia
force_realize(s::Proxy) =
  for b in current_backends()
    force_realize(b, s)
  end
```

### `ensure_ref(b, r)` — Wrap raw values into GenericRef

```julia
ensure_ref(b::Backend{K,T}, r::T) where {K,T} = NativeRef{K,T}(r)
ensure_ref(b::Backend{K,T}, rs::Vector{T1}) where {K,T,T1<:T} =
  length(rs) == 1 ?
    NativeRef{K,T}(rs[1]) :
    NativeRefs{K,T}(rs)
ensure_ref(b::Backend{K,T}, rs::Vector{Vector{T1}}) where {K,T,T1<:T} =
  ensure_ref(b, reduce(vcat, rs))
ensure_ref(b::Backend{K,T}, r::GenericRef{K,T}) where {K,T} = r
ensure_ref(b::Backend{K,T}, r::Any) where {K,T} = LocalRef{K,T}(r)
```

Dispatches on the type of `r`:
- Raw `T` value → `NativeRef{K,T}`
- `Vector{T}` with 1 element → `NativeRef{K,T}`
- `Vector{T}` with 2+ elements → `NativeRefs{K,T}`
- Nested `Vector{Vector{T}}` → flattened, then wrapped
- Already a `GenericRef{K,T}` → returned as-is
- Anything else → `LocalRef{K,T}`

There is also a special case for proxies used as surrogates:

```julia
ensure_ref(b::Backend{K,T}, v::Proxy) where {K,T} = ref(b, v)
```

This triggers realization of the surrogate proxy and returns its reference.

### `ref_value(b, r)` — Unwrap a GenericRef

```julia
ref_value(b::Backend{K,T}, a::Any) where {K,T} = a
ref_value(b::Backend, s::Proxy) = ref_value(b, ref(b, s))
ref_value(b::Backend{K,T}, r::NativeRef{K,T}) where {K,T} = r.value
ref_value(b::Backend{K,T}, r::NativeRefs{K,T}) where {K,T} = r.values
ref_value(b::Backend{K,T}, r::LocalRef{K,T}) where {K,T} = r.value
```

Extracts the raw value from a reference:
- `NativeRef` → single `T` value
- `NativeRefs` → `Vector{T}`
- `LocalRef` → the local value (type `Any`)
- `Proxy` → triggers `ref(b, s)` first, then unwraps
- Anything else → passed through unchanged

### `ref_values(b, r)` — Unwrap to a vector

```julia
ref_values(b::Backend{K,T}, r::NativeRef{K,T}) where {K,T} = T[r.value]
ref_values(b::Backend{K,T}, r::NativeRefs{K,T}) where {K,T} = r.values
ref_values(b::Backend, s::Proxy) = ref_values(b, ref(b, s))
ref_values(b::Backend{K,T}, ss::Proxies) where {K,T} =
  mapreduce(s->ref_values(b, ref(b, s)), vcat, ss, init=T[])
```

Always returns a `Vector{T}`, even for `NativeRef` (wraps in a single-element
vector). Used primarily by `b_delete_refs` to get all raw references for
deletion.

### `void_ref(b)` — Sentinel "no reference"

```julia
void_ref(b::MyBackend) = -1  # example
```

Each backend defines this to return a raw value of type `T` that represents
"not realized". **Must return a raw `T` value, not wrapped in `NativeRef`.**

### `reset_ref(b, s)` — Invalidate cached reference

```julia
reset_ref(b::Backend, s::Proxy) =
  delete!(refs_storage(b, s), s)
```

Removes the proxy from the reference cache. The next call to `ref(b, s)` will
re-realize the proxy.

### `realized(b, s)` — Check if proxy is cached

```julia
realized(b::Backend, s::Proxy) =
  haskey(refs_storage(b, s), s)
```

Returns `true` if the proxy has a cached reference in the backend.

## 4. References Storage

**Defined in**: `Backends.jl:9-52`

### The `References{K,T}` struct

```julia
struct References{K,T}
  shapes::Dict{Shape, GenericRef{K,T}}
  materials::Dict{Material, GenericRef{K,T}}
  layers::Dict{Layer, GenericRef{K,T}}
  annotations::Dict{Annotation, GenericRef{K,T}}
  families::Dict{Family, GenericRef{K,T}}
  levels::Dict{Level, GenericRef{K,T}}
end
```

Six separate dictionaries, one per proxy category. Each maps a proxy instance
to its `GenericRef{K,T}`.

### `refs_storage(b, proxy)` — Dispatch to the correct dictionary

```julia
refs_storage(b::Backend, p::Shape) = shape_refs_storage(b)       # b.refs.shapes
refs_storage(b::Backend, p::Material) = material_refs_storage(b)  # b.refs.materials
refs_storage(b::Backend, p::Layer) = layer_refs_storage(b)        # b.refs.layers
refs_storage(b::Backend, p::Annotation) = annotation_refs_storage(b)
refs_storage(b::Backend, p::Family) = family_refs_storage(b)
refs_storage(b::Backend, p::Level) = level_refs_storage(b)
```

All backends must have a `refs::References{K,T}` field. The default
`shape_refs_storage(b)` accesses `b.refs.shapes`, etc.

## 5. `@defproxy` vs `@defshape` Realization

### `@defproxy`

**Defined in**: `Shapes.jl:515-567`

Used for non-shape proxies (layers, materials, families) and some special
shapes. Generates:

```julia
realize(b::Backend, s::AbstractTypeName) =
  b_<name>(b, ref_value(b, s.field1), ref_value(b, s.field2), ...)
```

Key behavior: each field is recursively resolved via `ref_value`, which may
trigger realization of nested proxies. For instance, if a material contains a
layer field, that layer will be realized first.

### `@defshape`

**Defined in**: `Shapes.jl:727-744`

Used for all concrete shapes (point, line, box, sphere, etc.). Internally calls
`@defproxy` first (to generate the struct, constructor, predicates, etc.), then
overrides the `realize` method:

```julia
realize(b::Backend, s::TypeName) =
  b_<name>(b, s.field1, s.field2, ..., material_ref(b, s))
```

Key differences from `@defproxy`:
- Fields are passed **directly** (`s.field1`), not wrapped in `ref_value`
- **`material_ref(b, s)`** is appended as the last argument

The material field (added by `@defshape` with a default based on the shape
dimensionality) is excluded from the field list passed to `b_<name>` and instead
resolved separately via `material_ref`.

### Default materials by shape dimensionality

`@defshape` selects a default material based on the supertype:
- `Shape0D` → `default_point_material()`
- `Shape1D` → `default_curve_material()`
- `Shape2D` → `default_surface_material()`
- `Shape3D` → `default_material()`
- `Annotation` → `default_annotation_material()`

## 6. Material Realization

**Defined in**: `Shapes.jl:687-688`

```julia
material_ref(b::Backend, m::Material) = ref_value(b, m)
material_ref(b::Backend, s::Shape) = material_ref(b, s.material)
```

`material_ref` extracts the material from a shape and resolves it to a backend
reference. Since `ref_value(b, m::Proxy)` calls `ref(b, m)`, this triggers
lazy realization of the material if it hasn't been realized yet.

### `set_on!` (backend-specific material data)

```julia
set_on!(tb::Type{<:Backend}, proxy, ref) =
  begin
    proxy.data(tb, ref)
    for b in current_backend()
      if b isa tb
        reset_ref(b, proxy)
      end
    end
    proxy
  end
```

This sets backend-specific data on a material proxy and invalidates any cached
reference so the material will be re-realized with the new data. The public API
for this is `set_material`:

```julia
set_material(BackendType, material_proxy, backend_specific_data)
```

Note: `set_material` takes a backend **type** (not instance), e.g.,
`set_material(POVRay, my_material, povray_spec)`.

## 7. Transaction System

**Defined in**: `Shapes.jl:252-314`

### Transaction types

```julia
abstract type Transaction end

struct AutoCommitTransaction <: Transaction end

struct ManualCommitTransaction <: Transaction
  proxies::Vector{<:Proxy}
end
```

Each backend has a `transaction` parameter that holds the current transaction
type constructor (either `AutoCommitTransaction` or
`ManualCommitTransaction`).

### `maybe_realize` dispatch

```julia
maybe_realize(b, s) = maybe_realize(current_transaction(b)(), b, s)
```

- **`AutoCommitTransaction`**: realizes immediately if not already realized
  ```julia
  maybe_realize(::AutoCommitTransaction, b, s) =
    if ! realized(b, s)
      force_realize(b, s)
    end
  ```

- **`ManualCommitTransaction`**: defers realization, pushes to batch
  ```julia
  maybe_realize(t::ManualCommitTransaction, b, s) =
    push!(t.proxies, s)
  ```

### Commit and abort

```julia
commit_transaction(b, t::ManualCommitTransaction) =
  while !isempty(t.proxies)
    let s = popfirst!(t.proxies)
      if ! realized(b, s)
        force_realize(b, s)
      end
    end
  end

abort_transaction(b, t::ManualCommitTransaction) =
  empty!(t.proxies)
```

### `with_transaction` helper

```julia
with_transaction(f, b) =
  with(current_transaction(b), ensure_manual_commit_transaction(b)) do
    start_transaction(b)
    try
      f()
      commit_transaction(b)
    catch e
      abort_transaction(b)
      rethrow(e)
    end
  end
```

Temporarily switches to manual commit, runs `f()`, and commits (or aborts on
error).

### `LazyBackend`

```julia
abstract type LazyBackend{K,T} <: Backend{K,T} end
maybe_realize(b::LazyBackend, s) = save_shape!(b, s)
```

`LazyBackend` overrides `maybe_realize` entirely — instead of using the
transaction system, shapes are stored via `save_shape!`. This is used by
backends that accumulate shapes for later batch processing.

### IOBackend `realize_shapes`

IOBackend (used by TikZ, POVRay, Radiance) provides `realize_shapes` to
force-realize all accumulated shapes:

```julia
realize_shapes(b::IOBackend) =
  for s in b.shapes
    reset_ref(b, s)
    force_realize(b, s)
  end
```

Note that `reset_ref` is called before `force_realize` to ensure a fresh
realization. This is important for backends that write to files/buffers — the
shapes must be re-realized each time the output is regenerated.

## 8. Contract for Backend Implementors

### Required: `void_ref(b)`

Must return a raw value of type `T` (the backend's reference type). This value
is a sentinel meaning "no reference". **Do not wrap it in `NativeRef`.**

```julia
KhepriBase.void_ref(b::MyBackend) = -1  # for T = Int
```

### Required: `refs::References{K,T}` field

The backend struct must include a `refs::References{K,T}` field (either
directly or inherited via a base type like `IOBackend`). The default
`refs_storage` functions access `b.refs.*`.

### Required: `b_*` operations

Each `b_*` operation must return either:
- A single raw value of type `T` (will be wrapped in `NativeRef{K,T}`)
- A `Vector{T}` for multi-ref shapes (will be wrapped in `NativeRef` or
  `NativeRefs` depending on length)

`ensure_ref` handles the wrapping automatically.

### Optional: Override `refs_storage` accessors

If your backend doesn't use a field named `refs`, override
`shape_refs_storage`, `material_refs_storage`, etc.

## 10. Known Limitations and Design Notes

### Thebes: `void_ref` and shape refs are both `nothing`

Thebes uses `T = Union{Nothing, RGBA}` with `void_ref` returning `nothing`.
Since `b_*` operations also return `nothing`, every shape's ref is identical to
void_ref. This means `ref != void_ref` checks always evaluate to `false`, and
`realized()` always returns `true` after the first realization. In practice,
Thebes uses refs primarily for material tracking (where the ref is an `RGBA`
color), not shape identification.

### Makie: `T = Any` loses type safety

Makie defines `MakieId = Any`, which means `ensure_ref` will always match via
the `ensure_ref(b::Backend{K,T}, r::T)` method since everything is a subtype
of `Any`. The backend actually returns `Int` values from its `next_ref!()`
function. This works in practice but loses the type-level guarantee that refs
are integers.

### IOBackend: dummy return values by design

TikZ, POVRay, and Radiance are file-output backends. Their `b_*` operations
write to an IO buffer and return dummy values (typically `void_ref(b)`,
which is `-1`). This is by design — these backends don't need to track
individual shape references since their output is a complete file.

The `realize_shapes` function re-realizes all shapes from scratch each time
the output file is regenerated, so reference identity is irrelevant.
